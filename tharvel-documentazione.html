Documento Strategico Integrato

Admin panel universale per siti già esistenti.

Tharvel è pensato come alternativa professionale al ciclo WordPress + Elementor: consegna al cliente autonomia sui contenuti, mantenendo al developer controllo su Git, deploy, sicurezza e costi AI.

Non genera siti da zero. Adotta siti HTML, React, Next.js, Vue, Astro o WordPress e li rende gestibili tramite chat AI, preview live e pubblicazione controllata.
  • Self-hosted
  • BYOK
  • Git-native
  • Multi-site

Prodotto e Posizionamento

Tharvel si posiziona come admin panel universale installato dal developer sulla propria VPS e distribuito ai clienti insieme ai siti.

Problema che risolve

Il cliente vuole aggiornare testi, prezzi, immagini o sezioni senza aprire ticket. WordPress + Elementor risolve l'editing visuale, ma introduce plugin fragili, performance scadenti, sicurezza debole e scarso version control.

  • meno ticket manuali
  • meno stack WordPress
  • più autonomia cliente

Differenza chiave

Non è un CMS proprietario e non richiede di ricostruire il sito: si aggancia a repo esistenti via reverse proxy, lascia i repo cliente intatti e delega modifiche controllate a Pi Agent dal container centrale.

  • agnostico
  • Git-native
  • self-hosted

Confronto sintetico

Area WordPress + Elementor Tharvel

Architettura Target

La scelta centrale è co-locare l'admin nel sito e tenere intelligenza, segreti, Git e agent engine sulla VPS del developer.

Container centralizzato, repo cliente intatti

Decisione 2026-05-07: niente cartella _tharvel/ dentro i repo dei clienti. Il cliente accede da clientesito.com/tharveladmin, Nginx fa proxy verso il container Tharvel unico sulla VPS del developer, che risolve l'host nella tabella sites.

sito-cliente/
├── index.html      ← invariato
├── assets/
└── ...             ← zero codice Tharvel

Trade-off: singolo punto di guasto per tutti i pannelli admin (mitigato da healthcheck Coolify + auto-restart). Vantaggi: un solo codebase da aggiornare, repo cliente puliti, multi-tenancy reale.

VPS con due container logici

Tharvel App gestisce UI, auth, Git, upload, routing multi-tenant e WebSocket; Pi Agent Engine resta interno alla rete Docker e lavora sui volumi dei siti.

clientesito.com/tharveladmin
   → Nginx (X-Forwarded-Host)
   → Tharvel App :3000
   → DB lookup `sites` (domain → cwd_path)
   → Pi Agent Engine :8080
   → /var/www/siti-clienti/<dominio>/

Flusso Operativo

Il modello separa sempre bozza e produzione. L'agente lavora su preview; il sito live cambia solo dopo l'azione esplicita "Pubblica".

  1. Prompt o uploadIl cliente chiede una modifica in linguaggio naturale e può allegare immagini, PDF o asset.
  2. Pi Agent lavora sul branch previewModifica file sorgente, comprime immagini, aggiorna contenuti e manda streaming live.
  3. Preview aggiornataL'iframe mostra il risultato sul branch preview prima di toccare la produzione.
  4. Auto-commit post-turnDopo un turn riuscito e con file realmente modificati, Tharvel crea un commit leggibile su preview.
  5. Pubblicazione esplicitaClick su "Pubblica" → merge preview in main/master → push origin → webhook Coolify → deploy.
Policy chiusa 2026-05-10: auto-commit sì su preview dopo ogni turn riuscito; auto-push no. Il push avviene solo dentro il flusso "Pubblica".

Modello Sicurezza

La roadmap sicurezza parte da un principio netto: il prompt non è una difesa. La sicurezza deve essere imposta dal runtime, dal filesystem, dalla rete e dall'identità.

Filesystem

Rischio: un agente legge o scrive fuori dal sito. Difesa target: namespace bwrap che vede solo /work scrivibile.

Esecuzione

Rischio: shell arbitraria, installazioni, script malevoli. Difesa target: bash custom allowlisted dentro sandbox.

Network

Rischio: esfiltrazione o movimento laterale. Difesa target: default-deny host su reti private + binari minimi.

Identity / routing

Il client non deve mai scegliere il sito con una query string manipolabile. Il server valida un JWT firmato che contiene slug e role, poi ignora valori incompatibili.

Prompt injection

File del sito possono contenere istruzioni ostili. Con bwrap, anche se l'agente prova a leggere /etc/passwd o altri path, il namespace non li espone.

Backup e Restore

Il piano backup completa la sicurezza: non basta prevenire incidenti, bisogna poter ripristinare DB, sorgenti, upload e segreti con RTO/RPO misurati.

Tool e destinazione

Backup con restic, cifrato, deduplicato e incrementale. Destinazione scelta: OVH Object Storage Standard S3-compatible, region EU GRA o SBG.

Retention

Schema: 7 daily, 4 weekly, 6 monthly, 1 yearly. In Beta privata la frequenza è daily; a regime si rivaluta ogni 4 ore.

Restore drill completo

  • Provisioning VPS staging.
  • Installazione restic, git, node, npm, sqlite3.
  • restic check e verifica snapshot recente.
  • Restore latest su /restore.
  • PRAGMA integrity_check sul DB.
  • Build sito campione e smoke test Tharvel.

Target Beta

RTO: 4 ore. RPO: 24 ore. Il drill completo va fatto prima della Beta e ogni mese durante la Beta; il drill veloce deve stare sotto i 15 minuti.

Regola d'oro: la passphrase restic è il single point of recovery. Va custodita fuori dalla VPS, in password manager e copia fisica sicura.

Rischi e Prossime Mosse

I rischi principali sono già identificati nei documenti. La priorità è trasformarli in gate operativi verificabili.

Rischio 1 - Agent senza confinamento reale
Prima di clienti reali, implementare bwrap come fondazione tecnica: mount del solo sito come scrivibile, runtime read-only, binari minimi, deploy key per sito e test su VPS reale.
Rischio 2 - Routing multi-tenant manipolabile
Implementare Strato 4: login, JWT firmato con slug e role, validazione WebSocket e sidebar siti accessibile solo agli admin.
Rischio 3 - Backup non testato
Schedulare restic daily e completare un restore drill prima della Beta. Registrare RTO, RPO e problemi in un file dedicato ai drill.
Rischio 4 - Segreti recuperabili solo dalla VPS
Separare la passphrase restic dalla VPS, custodirla in password manager e copia fisica. Le credenziali S3 possono stare in backup cifrato, ma non la passphrase.
